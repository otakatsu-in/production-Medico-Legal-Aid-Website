# AI Context & Hostinger Deployment Knowledge Base

This document contains accumulated knowledge, architectural decisions, and deployment fixes for the MedicoLegalAid website. **Any AI assistant working on this repository should read this document first to understand the current state and prevent regression of deployment fixes.**

## 1. Monorepo Build Setup
The project is a `pnpm` workspace containing a Next.js frontend (`medicolegalaid-next`).
- **Hostinger Build Issue:** Hostinger's background deployment servers do not always have `pnpm` installed globally, which causes `pnpm: command not found` errors if you rely on it.
- **The Fix:** The root `package.json` build command was updated to use `npx pnpm`:
  `"build": "npx pnpm -F medicolegalaid-next run build"`
  This ensures `npx` automatically downloads `pnpm` on the fly and specifically builds only the frontend, ignoring other workspace artifacts.
- **pnpm Install Failure:** Modern pnpm versions block `postinstall` scripts for security reasons. If the build fails with `[ERR_PNPM_IGNORED_BUILDS]`, ensure `pnpm.onlyBuiltDependencies` is defined in `package.json` to allow packages like `esbuild` and `sharp` to build their binaries.

## 2. Next.js Routing & Hostinger CDN Conflicts (404/503 Errors)
- **The Problem:** Using Next.js `output: 'standalone'` causes severe conflicts with Hostinger's LiteSpeed CDN. The CDN intercepts requests for `/_next/static/*` and tries to find them at the root, but standalone mode buries them deep in `.next/standalone/...`. This results in client-side crashes and `404 Not Found` errors for Javascript chunks.
- **The Fix:** 
  1. We completely disabled `output: 'standalone'` in `next.config.ts`.
  2. We wrote a custom, highly robust `server.js` file located at `medicolegalaid-next/server.js`. This script programmatically boots the core Next.js routing engine, bypassing Hostinger's static asset interception completely.
  3. **Critical Path Context (503 Error):** Hostinger launches the Node server from the *workspace root*, not the Next.js directory. This caused a 503 crash because `require('next')` couldn't find the modules. We added `process.chdir(__dirname);` to the very top of `server.js` to strictly lock the working directory to `medicolegalaid-next` before any Next.js code runs.
- **Hostinger Config:** The "Entry file" in Hostinger must always be set to: `medicolegalaid-next/server.js`.

## 3. Post-Deployment Caching (ChunkLoadError)
- **The Problem:** Immediately after a new deployment, the website often crashes with a `ChunkLoadError` or 404s for Javascript files.
- **The Cause:** Hostinger/Cloudflare caches the *old* HTML page. When a new deployment occurs, new JS files with new hashes are generated. The old cached HTML asks for old JS hashes that have been deleted.
- **The Fix:** This is not a code error. It requires a hard browser refresh (`Cmd+Shift+R`) or flushing the cache in the Hostinger/Cloudflare dashboard. Do not alter the codebase to "fix" this.

## 4. Performance Optimizations (PageSpeed)
- **Images:** We removed `unoptimized: true` from `next.config.ts` so Next.js automatic image optimization (compression, WebP conversion, resizing) works correctly. All heavy images (like the hero portrait) must use the Next.js `<Image>` component instead of standard `<img>` tags to save bandwidth.
- **Third-Party Iframes:** The Vimeo player iframe heavily penalizes the Largest Contentful Paint (LCP) and blocks the main thread. It must always have `loading="lazy"` applied so it doesn't block the initial page render.

## 5. Security Widgets (Cloudflare Turnstile)
- If the Contact Form shows a "Bot verification failed" error with a red exclamation mark, it means the current domain is not whitelisted in the Cloudflare Turnstile dashboard.
- Any new temporary deployment domains (e.g., Hostinger preview URLs) must be added to the "Hostname Management" list in Cloudflare Turnstile for the widget to load.
